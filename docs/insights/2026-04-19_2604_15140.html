## 📖 背景与动机

在信息爆炸的时代，问答系统是用户获取知识的核心渠道。然而，现有研究多关注答案的「正确性」，忽视了「回答方式」的艺术性与策略性。人类在不同社区（如Stack Overflow、医学论坛、Reddit各子类）回答问题时，会根据社区规范、问题类型和用户期望采用截然不同的修辞策略——有时直接给出答案，有时先澄清问题，有时分享类似经历。这种策略多样性是人类沟通智慧的体现。

令人担忧的是，当前LLM的回答往往呈现「千篇一律」的风格：无论问题来自哪个领域、面向何种受众，GPT-4等模型都倾向于提供全面但冗长的回答。这种缺乏策略灵活性的表现，可能正是用户感知「AI味」浓重的根源。本研究正是要量化这种差异，并探索其背后的机制。


**关键要点：**

- 人类回答者会根据社区规范调整修辞策略，展现高度的情境适应性
- 当前LLM在问答场景中缺乏策略多样性，回答模式趋于同质化
- 修辞策略的多样性是区分高质量与低质量回答的重要维度，但尚未被充分研究

## 💡 核心方法

### 方法概述

DISCOTRACE将答案建模为「话语行为序列」，每一步行为都关联到对问题的特定解读，通过修辞结构理论(RST)提供话语组织框架。


### 详细设计

方法包含三个核心模块：

**1. 问题解读生成**：对每个问题，系统性地生成多个可能的解读（Interpretation）。例如「如何减肥？」可解读为：寻求具体方法、寻求动机建议、询问科学原理等。这些解读构成后续分析的基础。

**2. RST话语解析**：使用修辞结构理论对答案进行树形结构解析。RST将文本组织为核心关系（nucleus）和卫星关系（satellite），如「原因-结果」「问题-解决方案」等。这提供了基本的话语组织骨架。

**3. 话语行为标注**：在RST树的基础上，为每个叶节点标注具体的话语行为（Discourse Act）。论文定义了五大行为家族：
- **背景提供**：补充相关背景信息
- **答案传递**：直接回答问题的内容
- **证据支撑**：提供数据、研究、案例支持
- **确认理解**：回应问题中的前提或假设
- **元评论**：对回答本身或问题进行评价

最终，每个答案被表示为一个序列，格式为：(问题解读 → 话语行为)的配对序列。例如：
`(如何X → 确认理解) → (解读A → 答案传递) → (解读B → 证据支撑)`

这种表示法既捕获了「说什么」（通过问题解读），也捕获了「怎么说」（通过话语行为），为策略分析提供了统一的数学框架。

### 🔧 关键组件

| 组件 | 说明 |
|------|------|
| 问题解读生成器 | 基于prompt工程或小型模型，穷举式生成问题的多种合理解读，为每个答案建立「关注点-解读」的映射关系 |
| RST话语解析器 | 使用现有NLP工具（如SPADE）将答案切分为基本话语单元，并标注单元间的修辞关系，构建树形话语结构 |
| 话语行为标注系统 | 在半监督模式下，标注话语单元对应的行为类型（背景/答案/证据/确认/元评论），确保跨标注者一致性 |

class QuestionInterpretation:

        self.text = text        # 原始问题
        self.angle = angle      # 解读角度，如"方法"、"动机"、"原理"

class RSTNode:
    """RST树节点：修辞结构理论解析"""
    def __init__(self, relation_type, nucleus, satellite=None):
        self.relation = relation_type  # 关系类型：原因-结果、问题-解决方案等
        self.nucleus = nucleus         # 核心部分
        self.satellite = satellite     # 卫星部分（可选）
        self.children = []             # 子节点列表

class DiscourseAct:
    """话语行为标注"""
    ACT_TYPES = {
        "BG": "背景提供",      # 补充相关背景信息
        "AT": "答案传递",      # 直接回答问题
        "EV": "证据支撑",      # 提供数据、研究、案例
        "CK": "确认理解",      # 回应问题前提或假设
        "MC": "元评论"         # 对回答或问题进行评价
    }

# ---------------- 模块1: 问题解读生成 ----------------

    伪代码：根据问题生成多种可能解读

        ("障碍类", "了解减肥可能遇到的困难")

    interpretations = []
    for angle, description in interpretation_angles:
        # 实际应用中可用 LLM 或规则生成

        interpretations.append(interp)

    """
    伪代码：使用RST修辞结构理论解析答案
    输出: RSTNode 树形结构

    # 实际应用中需使用NLP解析器
    
    root = RSTNode("问题-解决方案", nucleus=None)
    
    # 主干：核心答案
    main_claim = RSTNode("主张", nucleus="减肥需要控制饮食加运动")
    
    # 支撑：背景信息
    bg_node = RSTNode("背景", nucleus="根据研究显示...")
    
    # 支撑：证据
    evidence_node = RSTNode("证据", nucleus="一项500人的研究...")
    
    # 组合关系
    main_claim.children = [bg_node, evidence_node]
    root.nucleus = main_claim
    
    return root

def extract_rst_leaves(rst_tree):
    """
    伪代码：从RST树提取所有叶节点
    """
    leaves = []
    def traverse(node):
        if not node.children:
            leaves.append(node)
        else:
            for child in node.children:
                traverse(child)
    traverse(rst_tree)
    return leaves

# ---------------- 模块3: 话语行为标注 ----------------
def annotate_discourse_act(rst_node, matched_interpretation):
    """
    伪代码：根据RST节点类型标注话语行为

    act_mapping = {

        "主张": "AT",          # 答案传递

    # 简单规则映射（实际需更复杂逻辑）

# ---------------- 主流程：生成序列表示 ----------------

    伪代码：完整流程 - 生成 (问题解读 → 话语行为) 序列

**主要结果**：

实验揭示了三个关键发现：①人类社区策略熵显著高于LLM(p<0.01)，表明人类回答具有更丰富的策略变化；②LLM被提示「模仿某社区风格」时，仅能部分迁移词汇和格式偏好，但核心话语行为序列模式几乎不变；③LLM系统性偏好「广度优先」，尝试覆盖更多问题解读，而人类更倾向「深度优先」，选择性地深入某一解读。


**主要发现：**

- ✅ 不同人类社区展现出显著差异的修辞策略偏好，如技术社区倾向直接答案型，情感社区偏好共情-建议型
- ✅ 即使使用few-shot prompting要求模仿特定社区，LLM仍难以习得人类策略的多样性和选择性
- ✅ LLM存在「过度泛化」倾向：总是尝试回答所有可能解读，而非像人类那样有策略地取舍

## ❓ 常见问题

**Q：为什么LLM会倾向于「广度优先」而不是像人类那样「深度优先」？**

A：这可能源于LLM的训练目标——最大化答案的「全面性」和「有帮助性」被隐式定义为覆盖更多知识点。同时，RLHF奖励模型倾向于给全面答案更高分，忽略了人类回答中「选择性沉默」（不回答某些解读）的智慧。


**Q：这项研究对普通用户有什么实际意义？**

A：用户在日常使用LLM时，常感觉「AI回答太空洞」或「不够像人在说话」。本研究揭示这一现象的深层原因：LLM缺乏策略选择性。理解这点后，用户可以通过更精确的prompt（如「请用Stack Overflow风格回答」）来引导LLM调整策略，获得更接近人类社区期望的回答。


**Q：如果想在自己的项目中使用DISCOTRACE，需要准备什么？**

A：需要：①RST解析工具（如Stanford RST Parser）；②已标注的话语行为数据集（论文提供了标注指南）；③计算资源进行大规模序列分析。对于垂直领域应用，建议先在小数据集上验证标注一致性，再扩展到全量数据。
