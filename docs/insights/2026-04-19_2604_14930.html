## 📖 背景与动机

当前大语言模型(LLM)在处理复杂任务时面临两大核心挑战：一是上下文窗口长度限制导致无法一次性加载全部信息，二是传统RAG(检索增强生成)方法将原始文本作为静态只读块处理，造成大量冗余计算。当Agent执行多步推理任务时，每个步骤都需要重新扫描上下文以定位相关信息，这不仅效率低下，还容易引入噪声。研究者意识到，计算机架构中CPU缓存的思想或许可以迁移到LLM推理场景——通过预先提取和组织关键信息，实现按需快速访问，从而避免重复的全文扫描开销。


**关键要点：**

- LLM上下文窗口有限，无法一次性处理超长文档的全部信息
- 传统RAG将原始文本作为静态只读块，缺少动态信息组织和复用机制
- 多步推理中相邻步骤存在大量重复信息访问，造成计算资源浪费
- 计算机架构中缓存思想为解决该问题提供了理论启发

## 💡 核心方法

### 方法概述

IE-as-Cache框架将信息抽取模块嵌入Agent推理流程，充当动态缓存角色。系统为每个推理步骤生成查询条件化的抽取模式(Query-conditioned Schema)，从输入文本中提取与当前步骤相关的结构化信息，并以缓存形式存储供后续步骤复用。


### 详细设计

该框架的核心设计包含三个关键机制。首先是查询条件化抽取模式(Query-conditioned Extraction Schema)：不同于传统的固定Schema抽取，该方法根据当前推理步骤的查询(query)动态生成针对性的抽取模式qt，qt包含该步骤的特定信息焦点(Specific Focus)，明确指示需要从上下文中提取什么信息。这种设计确保每次抽取都是有目的、高相关性的信息提取。其次是动态缓存管理机制：提取的结构化信息E = F_extract(text, qt)被存储在一个模拟CPU缓存结构的认知记忆中，每个缓存条目都与特定的查询条件关联，支持快速索引和召回。第三是层次化记忆架构：框架构建了与计算机内存层次结构(CPU缓存→主存→磁盘)相对应的认知记忆层次，用户可以根据精度和速度的权衡选择不同层次的存储和检索策略。


### 🔧 关键组件

| 组件 | 说明 |
|------|------|
| Query-conditioned Schema Generator | 根据当前推理步骤的查询动态生成抽取模式，包含step-specific focus，明确指定需要提取的信息类型和范围 |
| Cognitive Cache Memory | 模拟CPU缓存的层次化记忆结构，存储抽取后的结构化信息，支持快速索引和按需召回，与主存形成层次化管理 |
| IE Engine (F_extract) | 核心信息抽取执行器，接收Schema和原始文本，输出结构化的抽取结果E，支持多种抽取范式 |

"""
简化的认知记忆框架示例
核心：查询条件化抽取 + 动态缓存 + 层次化记忆
"""

# ============ 1. 查询条件化抽取模式 ============
class QueryConditionedExtractor:
    """根据查询动态生成抽取模式"""
    
    def generate_schema(self, query, context):
        """根据query生成针对性的抽取模式qt"""
        # 分析查询意图，生成特定的信息焦点
        focus = self._identify_specific_focus(query)
        schema = {
            'target_info': focus,      # 信息焦点
            'constraints': query,       # 查询约束
            'extraction_rules': []      # 抽取规则
        }
        return schema
    
    def extract(self, text, query):
        """基于条件化模式抽取信息"""
        qt = self.generate_schema(query, text)  # 动态生成模式
        entities = self._apply_rules(text, qt)  # 应用抽取规则
        return entities


# ============ 2. 动态缓存管理机制 ============
class CognitiveCache:
    """模拟CPU缓存结构的认知记忆"""
    
    def __init__(self, capacity=100):
        self.cache = {}           # 缓存存储
        self.capacity = capacity
        self.hit_count = 0
    
    def store(self, query_key, extracted_info):
        """存储与查询条件关联的信息"""
        if len(self.cache) >= self.capacity:
            self._evict_lru()     # LRU淘汰策略
        self.cache[query_key] = {
            'data': extracted_info,
            'query': query_key,    # 关联的查询条件
            'access_time': 0
        }
    
    def retrieve(self, query_key):
        """快速索引和召回"""
        if query_key in self.cache:
            self.hit_count += 1
            self.cache[query_key]['access_time'] += 1
            return self.cache[query_key]['data']
        return None
    
    def _evict_lru(self):
        """最近最少使用淘汰"""
        lru_key = min(self.cache.keys(), 
                      key=lambda k: self.cache[k]['access_time'])
        del self.cache[lru_key]


# ============ 3. 层次化记忆架构 ============
class HierarchicalMemory:
    """模拟计算机内存层次结构"""
    
    def __init__(self):
        self.l1_cache = CognitiveCache(capacity=50)      # CPU缓存 - 最高速
        self.l2_memory = {}                               # 主存 - 中速
        self.l3_storage = []                              # 磁盘 - 低速大容量
    
    def store(self, info, query, level='auto'):
        """分层存储，根据权衡选择层次"""
        if level == 'auto':
            # 根据信息重要性/访问频率自动选择层次
            level = self._select_tier(info)
        
        if level == 'l1':
            self.l1_cache.store(query, info)
        elif level == 'l2':
            self.l2_memory[query] = info
        else:
            self.l3_storage.append({'query': query, 'info': info})
    
    def retrieve(self, query, level='l1'):
        """分层检索"""
        if level == 'l1':
            return self.l1_cache.retrieve(query)
        elif level == 'l2':
            return self.l2_memory.get(query)
        else:
            # 慢速搜索
            for item in self.l3_storage:
                if item['query'] == query:
                    return item['info']
        return None
    
    def _select_tier(self, info):
        """根据精度/速度权衡选择存储层"""
        return 'l1'  # 简化：默认最高速


# ============ 主框架整合 ============
class CognitiveMemoryFramework:
    """认知记忆框架 - 整合三大机制"""
    
    def __init__(self):
        self.extractor = QueryConditionedExtractor()
        self.cache = CognitiveCache()
        self.hierarchy = HierarchicalMemory()
    
    def process(self, text, query):
        """完整处理流程"""
        # Step 1: 查询条件化抽取
        extracted_info = self.extractor.extract(text, query)
        
        # Step 2: 存入缓存
        self.cache.store(query, extracted_info)
        
        # Step 3: 分层存储
        self.hierarchy.store(extracted_info, query)
        
        return extracted_info
    
    def query(self, query):
        """快速查询 - 优先缓存"""
        # 尝试从缓存快速召回
        cached = self.cache.retrieve(query)
        if cached:
            return cached, 'cache_hit'
        return None, 'cache_miss'


# ============ 使用示例 ============
if __name__ == '__main__':
    framework = CognitiveMemoryFramework()
    
    text = "苹果公司成立于1976年，创始人是乔布斯..."
    query = "苹果公司的成立时间"
    
    # 处理文本
    result = framework.process(text, query)
    print(f"抽取结果: {result}")
    
    # 查询缓存
    cached_result, status = framework.query(query)
    print(f"缓存状态: {status}, 结果: {cached_result}")

## ❓ 常见问题

**Q：IE-as-Cache与传统RAG的本质区别是什么？**

A：传统RAG将原始文本作为静态块传递，每个推理步骤都需要重新检索和阅读相关内容；而IE-as-Cache通过预抽取和缓存机制，将信息以结构化形式存储，支持直接复用，大幅减少重复计算。


**Q：该方法适用于所有Agent任务吗？**

A：更适合涉及大量文本信息、需要多步推理的长程任务。对于简单的一次性问答或信息密集度低的任务，传统方法可能已经足够高效。


**Q：如何确定Query-conditioned Schema的质量？**

A：Schema质量取决于两个因素：query的表达清晰度和LLM对抽取指令的理解能力。建议通过few-shot示例或微调来提升Schema生成的稳定性。
