## 📖 背景与动机

随着大语言模型（LLM）在各行业的广泛应用，基于LLM的Agent系统（Agentic Workflows）正成为构建复杂AI应用的主流范式。这类工作流通过编排多个LLM和外部工具，能够完成单LLM无法胜任的复杂任务，如多步骤推理、代码生成、自主规划等。然而，将这类工作流部署到生产环境面临严峻挑战：首先，工作流中的LLM数量往往远超可用GPU数量，导致严重的GPU资源竞争；其次，工作流的执行图具有高度动态性，可能包含条件分支、并行扇出、数据依赖的递归等复杂控制流，这使得端到端延迟难以预测。传统的GPU调度方法主要针对单一模型优化，无法有效处理这种多模型、多分支的复杂场景。因此，如何在保证低延迟的同时实现高吞吐，成为制约Agent系统大规模落地的关键瓶颈。


**关键要点：**

- Agentic工作流通过多LLM编排实现复杂任务，但GPU资源有限导致过度订阅问题
- 工作流执行具有不可预测性：分支、扇出、递归等动态执行路径
- 传统调度方法无法适应多模型、非线性执行的复杂场景

## 💡 核心方法

### 方法概述

Scepsy是首个针对任意Agent工作流的GPU调度系统，其核心创新在于发现并利用LLM执行时间占比的稳定性：虽然工作流总执行时间波动大，但每个LLM占总时间的比例相对固定。基于这一洞察，系统通过profiling获取各LLM的统计特征，构建Aggregate LLM Pipeline来实现高效的GPU资源分配。


### 详细设计

Scepsy的调度流程包含三个关键阶段。首先是Profiling阶段：系统对工作流中的每个LLM在不同并行度（如batch size、tensor parallelism degree）下进行基准测试，记录其执行时间和资源占用。通过统计分析，得到每个LLM在整体执行时间中的占比分布（aggregate share）。这一占比在不同输入、不同执行路径下保持相对稳定，是后续调度的关键依据。第二阶段是Pipeline Construction：基于profiling得到的统计信息，系统构建Aggregate LLM Pipeline。这是一个逻辑抽象，将多个LLM的调度决策转化为流水线式的资源分配问题。对于beam search这类典型工作流（Figure 4展示了其构建过程），系统识别出encoder、decoder、scorer等核心组件，并根据各自的aggregate share分配GPU资源。第三阶段是Runtime Scheduling：在工作流执行过程中，系统根据预先计算的分配策略和实时监控的信息，动态调整GPU placement。特别地，Scepsy引入了topology-aware fractional GPU placement机制，考虑GPU之间的通信拓扑，将LLM放置在最优的GPU组合上，以最小化跨节点通信开销。当某个LLM的执行时间偏离预期时，系统会触发重调度，但受益于aggregate share的稳定性，调整范围有限，不会导致频繁抖动。

import random
from dataclasses import dataclass
from typing import Dict, List

# ============ 数据结构定义 ============
@dataclass
class LLMComponent:
    name: str
    execution_times: List[float]  # 不同并行度下的执行时间
    resource_usage: Dict[str, float]  # 资源占用

@dataclass
class AggregateInfo:
    share: float  # 执行时间占比
    optimal_parallelism: tuple  # 最优并行配置
    variance: float  # 时间方差（稳定性指标）

# ============ 第一阶段：Profiling ============
class Profiler:
    def profile_llm(self, component: LLMComponent) -> AggregateInfo:
        """对单个LLM进行基准测试，计算aggregate share"""
        # 遍历不同并行度配置进行测试
        best_time = float('inf')
        best_config = None
        
        for bs in [1, 2, 4, 8]:  # batch size
            for tp in [1, 2, 4]:  # tensor parallelism degree
                execution_time = self._benchmark(component, bs, tp)
                if execution_time < best_time:
                    best_time = execution_time
                    best_config = (bs, tp)
        
        # 计算aggregate share（执行时间占比）
        aggregate_share = best_time / sum(
            self._benchmark(c, 1, 1) for c in self.workflow_components
        )
        
        # 计算方差评估稳定性
        variance = self._calculate_variance(component.execution_times)
        
        return AggregateInfo(aggregate_share, best_config, variance)
    
    def _benchmark(self, component: LLMComponent, bs: int, tp: int) -> float:
        """模拟基准测试"""
        base_time = component.execution_times[0]
        return base_time / (bs * tp) + random.uniform(0, 0.1)
    
    def _calculate_variance(self, times: List[float]) -> float:
        """计算方差"""
        mean = sum(times) / len(times)
        return sum((t - mean) ** 2 for t in times) / len(times)

# ============ 第二阶段：Pipeline Construction ============
class PipelineConstructor:
    def build_aggregate_pipeline(self, components: List[LLMComponent], 
                                  aggregate_info: Dict[str, AggregateInfo]) -> Dict:
        """构建Aggregate LLM Pipeline"""
        # 按aggregate share分配GPU资源
        total_share = sum(info.share for info in aggregate_info.values())
        
        pipeline = {
            'stages': [],
            'gpu_allocation': {}
        }
        
        for comp in components:
            info = aggregate_info[comp.name]
            gpu_fraction = info.share / total_share
            
            pipeline['stages'].append({
                'component': comp.name,
                'aggregate_share': info.share,
                'gpu_fraction': gpu_fraction,
                'parallelism': info.optimal_parallelism
            })
            
            # 识别核心组件类型
            if 'encoder' in comp.name:
                pipeline['gpu_allocation'][comp.name] = 'encoder_gpu_pool'
            elif 'decoder' in comp.name:
                pipeline['gpu_allocation'][comp.name] = 'decoder_gpu_pool'
            elif 'scorer' in comp.name:
                pipeline['gpu_allocation'][comp.name] = 'scorer_gpu_pool'
        
        return pipeline

# ============ 第三阶段：Runtime Scheduling ============
class RuntimeScheduler:
    def __init__(self, gpu_topology: Dict):
        self.gpu_topology = gpu_topology  # GPU通信拓扑
    
    def schedule(self, pipeline: Dict, 
                 current_metrics: Dict[str, float]) -> Dict:
        """动态调度：考虑拓扑感知的fractional GPU placement"""
        new_placement = {}
        
        for stage in pipeline['stages']:
            comp_name = stage['component']
            expected_time = stage['aggregate_share']
            actual_time = current_metrics.get(comp_name, expected_time)
            
            # 检测偏差
            deviation = abs(actual_time - expected_time) / expected_time
            
            if deviation > 0.2:  # 触发重调度阈值
                # topology-aware GPU选择
                optimal_gpus = self._find_optimal_gpus(
                    comp_name, 
                    stage['gpu_fraction'],
                    stage['parallelism']
                )
                new_placement[comp_name] = {
                    'gpus': optimal_gpus,
                    'deviation': deviation,
                    'rescheduled': True
                }
            else:
                # 保持原有placement（aggregate share稳定性保证）
                new_placement[comp_name] = {
                    'gpus': pipeline['gpu_allocation'].get(comp_name),
                    'deviation': deviation,
                    'rescheduled': False
                }
        
        return new_placement
    
    def _find_optimal_gpus(self, comp_name: str, 
                           gpu_fraction: float, 
                           parallelism: tuple) -> List:
        """基于GPU拓扑寻找最优GPU组合"""
        # 模拟拓扑感知选择：优先选择同一节点内通信开销低的GPU
        num_gpus = int(gpu_fraction * 8)  # 假设总共8个GPU
        
        # 简化：选择连续的GPU（同一NUMA节点）
        return list(range(num_gpus))

# ============ 主调度流程示例 ============
def scepsy_scheduler(workflow_components: List[LLMComponent], 
                     gpu_topology: Dict):
    # --- Phase 1: Profiling ---
    profiler = Profiler()
    profiler.workflow_components = workflow_components
    
    aggregate_info = {}
    for comp in workflow_components:
        aggregate_info[comp.name] = profiler.profile_llm(comp)
        print(f"[Profiling] {comp.name}: share={aggregate_info[comp.name].share:.3f}")
    
    # --- Phase 2: Pipeline Construction ---
    constructor = PipelineConstructor()
    pipeline = constructor.build_aggregate_pipeline(workflow_components, aggregate_info)
    
    print(f"\n[Pipeline] Built {len(pipeline['stages'])} stages")
    for stage in pipeline['stages']:
        print(f"  {stage['component']}: {stage['gpu_fraction']*100:.1f}% GPU")
    
    # --- Phase 3: Runtime Scheduling ---
    scheduler = RuntimeScheduler(gpu_topology)
    current_metrics = {
        'encoder': 0.15,  # 模拟实时执行时间
        'decoder': 0.25,
        'scorer': 0.10
    }
    
    placement = scheduler.schedule(pipeline, current_metrics)
    
    print(f"\n[Scheduling] Runtime placement:")
    for comp, info in placement.items():
        status = "RESCHEDULED" if info['rescheduled'] else "stable"
        print(f"  {comp}: GPUs={info['gpus']} [{status}]")

# ============ 运行示例 ============
if __name__ == "__main__":
    # 模拟workflow components (beam search工作流)
    components = [
        LLMComponent("encoder", [0.15, 0.08, 0.05, 0.03]),
        LLMComponent("decoder", [0.25, 0.13, 0.07, 0.04]),
        LLMComponent("scorer", [0.10, 0.05, 0.03, 0.02]),
    ]
    
    # 模拟GPU拓扑 (8 GPUs, 2 nodes x 4 GPUs)
    gpu_topology = {
        'node0': [0, 1, 2, 3],
        'node1': [4, 5, 6, 7],
        'inter_node_latency': 100,  # 跨节点延迟
        'intra_node_latency': 5     # 同节点延迟
    }
    
    scepsy_scheduler(components, gpu_topology)

## ❓ 常见问题

**Q：为什么aggregate share会稳定？**

A：虽然输入数据不同导致绝对执行时间波动，但每个LLM的计算复杂度与输入长度的关系相对固定，因此各LLM占总时间的比例保持稳定。


**Q：Fractional GPU placement具体指什么？**

A：指将单个GPU的资源（如显存、计算单元）按比例分配给不同的LLM，而非传统的一对一绑定。例如一个LLM占用GPU的40%资源，另一个占用60%。


**Q：Scepsy与vLLM等现有推理系统是什么关系？**

A：Scepsy定位为更上层的调度层，可以整合现有推理引擎。它解决的是多LLM协作时的资源分配问题，而非单一LLM的推理优化。
