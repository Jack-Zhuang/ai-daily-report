## 📖 背景与动机

检索增强生成（RAG）已成为大语言模型落地的重要技术，通过引入外部知识解决模型幻觉和知识过时问题。然而实际应用中，检索返回的上下文往往存在噪声（无关文档）、不完整性（缺失关键证据）和异构性（文档格式、风格不一致）等问题。传统单生成流程难以有效协调这些矛盾证据，常导致回答质量不稳定。MASS-RAG正是针对这一痛点提出，通过多智能体分工机制，让不同智能体专注于特定证据处理任务，最终通过合成器整合多视角信息，显著提升系统在困难场景下的鲁棒性。


**关键要点：**

- RAG面临的核心挑战：检索结果常包含噪声、不完整或异构的上下文
- 单生成流程的局限：难以有效调和矛盾证据，容易被误导
- 解决思路：引入多智能体架构，分工处理证据的不同方面
- 核心假设：多智能体的中间视图能让模型在生成前充分比较和整合信息

## 💡 核心方法

### 方法概述

MASS-RAG采用多智能体协作架构，包含三个核心处理智能体（摘要、抽取、推理）和一个专用合成器。检索文档首先经过多智能体处理生成多视角中间表示，再由合成器整合产生最终答案。


### 详细设计

系统首先接收用户查询和检索返回的最多20个文档。三个专业化智能体并行工作：

1. **摘要智能体**：对每个检索文档生成压缩摘要，识别核心论点和关键信息，帮助快速判断文档相关性。

2. **抽取智能体**：从文档中精确提取与查询直接相关的证据片段，减少无关信息的干扰。

3. **推理智能体**：对证据进行逻辑分析和推理，识别因果关系、对比关系等深层语义联系。

三个智能体的输出传递给**合成器**，合成器整合多视角信息，处理可能的矛盾和互补关系，生成最终答案。这种设计的关键优势在于：即使某个智能体的输出有误或存在噪声，其他智能体的独立视角可以作为校正参考，大大提高了系统的鲁棒性。

### 🔧 关键组件

| 组件 | 说明 |
|------|------|
| 摘要智能体(Summarizer) | 对每个检索文档生成结构化摘要，压缩冗余信息，提取文档主旨。帮助系统快速判断文档整体相关性，减少后续处理的无用功。 |
| 抽取智能体(Extractor) | 精确提取与查询直接相关的证据片段，过滤修饰性文字。专注事实性信息提取，为推理阶段提供干净的结构化输入。 |
| 推理智能体(Reasoner) | 对证据进行深度语义分析，识别论点间的逻辑关系、因果链条、对比关系。揭示表面信息下的深层语义联系。 |
| 合成器(Synthesizer) | 整合三个智能体的多视角输出，处理矛盾信息，综合互补观点。生成连贯、一致、信息完整的最终答案，是系统的决策中枢。 |

import random
from typing import List, Dict

# ============ 模拟数据 ============
QUERY = "人工智能对就业市场的影响"
RETRIEVED_DOCS = [f"文档{i}" for i in range(20)]  # 最多20个文档

# ============ 三个智能体 ============

class SummaryAgent:
    """摘要智能体：生成压缩摘要，识别核心论点"""
    
    def process(self, doc: str) -> str:
        # 模拟：生成文档摘要
        return f"摘要({doc}): 核心论点是..."

class ExtractionAgent:
    """抽取智能体：提取与查询相关的证据片段"""
    
    def process(self, doc: str, query: str) -> str:
        # 模拟：精确提取相关证据
        return f"证据({doc}): 提取的相关片段..."

class ReasoningAgent:
    """推理智能体：逻辑分析和推理，识别深层语义"""
    
    def process(self, evidence: str) -> str:
        # 模拟：因果分析、对比分析
        return f"推理结果({evidence}): 因果关系/对比关系..."

# ============ 合成器 ============

class Synthesizer:
    """合成器：整合多视角信息，处理矛盾和互补"""
    
    def synthesize(self, summaries: List, evidences: List, 
                    reasoning: List) -> str:
        # 模拟：整合信息，生成最终答案
        return f"最终答案：基于{len(summaries)}个视角的综合分析结果"

# ============ 主流程 ============

def main():
    # 初始化智能体
    summary_agent = SummaryAgent()
    extraction_agent = ExtractionAgent()
    reasoning_agent = ReasoningAgent()
    synthesizer = Synthesizer()
    
    # 存储输出
    all_summaries = []
    all_evidences = []
    all_reasonings = []
    
    # 遍历每个文档（可并行优化）
    for doc in RETRIEVED_DOCS:
        
        # 1. 摘要智能体
        summary = summary_agent.process(doc)
        all_summaries.append(summary)
        
        # 2. 抽取智能体
        evidence = extraction_agent.process(doc, QUERY)
        all_evidences.append(evidence)
        
        # 3. 推理智能体（基于证据）
        reasoning = reasoning_agent.process(evidence)
        all_reasonings.append(reasoning)
    
    # 4. 合成器整合
    final_answer = synthesizer.synthesize(
        all_summaries, 
        all_evidences, 
        all_reasonings
    )
    
    return final_answer

# ============ 运行 ============
if __name__ == "__main__":
    result = main()
    print(result)

## 🔬 实验结果

**数据集**：PopQA、TriviaQA等四个标准RAG评测基准（涵盖开放域问答、多跳推理等任务类型）

**评价指标**：答案准确率、Evidence Coverage Rate (ECR)、各基线对比指标

**主要结果**：

MASS-RAG在四个基准上均超越强基线，相比Self-RAG等先进方法有明显提升。消融实验表明三个智能体的协同工作对性能贡献显著，禁用任一智能体都会导致性能下降。ECR分析显示多智能体架构能更全面覆盖关键证据，尤其在证据分散或隐含的场景下优势明显。


**主要发现：**

- ✅ 多智能体分工机制有效提升系统在噪声检索场景下的鲁棒性
- ✅ 三个智能体的互补作用：摘要提供全局视角，抽取保证精确性，推理挖掘深层联系
- ✅ 合成器是关键组件，禁用Answer Agent后仍能通过合成器产生有效输出
- ✅ 在证据稀缺或不完整的情况下，性能提升尤为显著

## ❓ 常见问题

**Q：MASS-RAG相比传统RAG的主要优势是什么？**

A：传统RAG的单一生成流程在面对噪声、不完整或异构的检索结果时容易被误导。MASS-RAG通过三个专业化智能体并行处理证据，产生多视角中间表示，再由合成器整合。这种设计让系统能够：1) 多角度审视证据，降低单一视角的偏见；2) 智能体间形成互补和校正机制；3) 在证据稀缺时仍能综合有效信息。实验证明在四个基准上均有显著提升。


**Q：实现MASS-RAG需要哪些资源？**

A：核心需要三个独立的大模型实例（或一个模型三次调用），加上一个合成器模型。模型规模建议7B以上以保证各智能体任务的质量。延迟方面，三智能体可并行处理，实际增加的时间约为单智能体的1.2-1.5倍（而非三倍）。部署时可采用批量处理和异步调用进一步优化。


**Q：如果某个智能体产生错误输出会影响整体性能吗？**

A：相比单一流水线，多智能体架构的容错性更强。因为合成器接收三个独立视角的输入，当某个智能体出错时，其他两个智能体的正确信息可以作为补偿。论文的消融实验也验证了这点——禁用任一智能体都会导致性能下降，说明各智能体都有正向贡献，但也说明系统不会因为单一智能体失败而完全崩溃。


**Q：MASS-RAG适合哪些实际应用场景？**

A：最适合以下场景：1) 检索结果质量不可控的多源异构数据场景；2) 对答案准确性要求高的专业领域（法律、医疗、金融）；3) 需要综合多份文档才能回答的复杂查询；4) 用户问题模糊需要系统自主判断相关性的场景。不建议用于简单事实查询（开销不划算）或实时性要求极高的场景。
