📖 背景与动机

序列推荐系统旨在基于用户的历史行为序列预测下一时刻可能感兴趣的物品。传统方法如SASRec、BERT4Rec等虽有一定效果，但在用户意图理解和冷启动场景下表现有限。与此同时，大语言模型(LLM)展现了强大的语义理解和推理能力，能够从用户画像、行为描述中推断出深层偏好。

然而，直接将LLM部署于推荐场景面临巨大挑战：计算资源消耗大、推理延迟高、难以实时更新。知识蒸馏成为折中方案，但现有方法往往忽视了两个关键问题：(1)用户中心知识的重要性——推荐本质是理解和服务用户；(2)蒸馏过程中损失尺度不平衡导致的优化困难。

本文正是针对这两个痛点，提出了用户中心的知识蒸馏框架，兼顾效果与效率。

关键要点：
  • 序列推荐需要理解用户行为的时序依赖和潜在意图
  • LLM具备强大的语义推理能力，但部署成本高昂
  • 知识蒸馏是平衡效果与效率的可行路径
  • 用户中心视角和损失平衡是蒸馏成功的关键

💡 核心方法

方法概述

提出用户中心知识蒸馏框架(UCK-Distill)，将LLM的语义知识通过蒸馏技术迁移到轻量级Transformer推荐模型，核心是通过stop-gradient动态调整蒸馏强度。

详细设计

框架包含三个核心阶段：

(1) 用户画像构建：利用LLM分析用户历史交互序列，自动生成包含偏好特征、兴趣标签、评分倾向的结构化用户画像。LLM能够区分用户的高度评价物品和低评价物品，从而识别真正的偏好特征。

(2) 知识转移：设计用户中心的蒸馏损失函数，将LLM产生的软标签(Soft Labels)作为监督信号传递给学生模型。这里涉及的关键挑战是蒸馏损失(Ldistill)通常远小于模型损失(Lmodel)，导致优化偏向主损失而忽视蒸馏知识。

(3) 动态平衡机制：引入动态缩放因子β来解决损失尺度不平衡问题。公式中通过detach操作确保β只用于损失加权而不产生梯度回传，实现自适应的蒸馏强度调节。训练过程中β会随epoch动态演化，使模型在初期专注学习基本表示，后期强化知识融合。

学生模型采用标准Transformer编码器架构，处理物品ID序列，接收来自教师(LLM)的双重监督：用户画像级语义指导和物品级偏好预测。

class DynamicDistillationLoss(nn.Module):

"""

用户中心的蒸馏损失 + 动态缩放因子β

解决Ldistill << Lmodel 的尺度不平衡问题

"""

def __init__(self, initial_beta=1.0):

super().__init__()

self.beta = initial_beta # 动态缩放因子

def compute_distill_loss(self, student_output, teacher_soft_labels):

"""

计算蒸馏损失：学生输出 vs LLM软标签

"""

# 软标签通常为概率分布，用KL散度衡量差异

distill_loss = F.kl_div(

F.log_softmax(student_output, dim=-1),

teacher_soft_labels,

reduction='batchmean'

)

return distill_loss

def update_beta(self, epoch, total_epochs):

"""

动态演化β：

- 初期: β较小，专注学习基本表示

- 后期: β较大，强化知识融合

"""

# detach确保β只用于加权，不产生梯度回传

progress = epoch / total_epochs

self.beta = self.beta * (1 + progress) # 简化的线性增长策略

return self.beta.detach()

class DistillationTraining:

def __init__(self, student_model, teacher_llm, profile_builder):

self.student = student_model

self.teacher = teacher_llm

self.profile_builder = profile_builder

self.distill_loss_fn = DynamicDistillationLoss(initial_beta=0.1)

self.optimizer = torch.optim.Adam(student_model.parameters(), lr=1e-4)

def training_step(self, batch, epoch):

user_ids, item_seqs, ratings = batch

# ---------- Step 1: 教师端生成软标签 ----------

# 1.1 构建用户画像

profiles = []

soft_labels = []

for user_id, item_seq in zip(user_ids, item_seqs):

# 获取用户历史交互

history = self._get_user_history(user_id)

# LLM生成用户画像

profile = self.profile_builder.build_profile(user_id, history)

profiles.append(profile)

# LLM生成物品级偏好预测（软标签）

soft_label = self.teacher.predict_preference(user_id, item_seq)

soft_labels.append(soft_label)

teacher_profile = torch.stack([p['embedding'] for p in profiles])

teacher_preference = torch.stack(soft_labels)

# ---------- Step 2: 学生端前向传播 ----------

student_profile_pred, student_pref_pred = self.student(item_seqs)

# ---------- Step 3: 计算各项损失 ----------

# 3.1 模型主损失（物品级预测损失）

model_loss = F.cross_entropy(student_pref_pred.view(-1, student_pref_pred.size(-1)),

ratings.view(-1))

# 3.2 画像级蒸馏损失

profile_loss = F.mse_loss(student_profile_pred, teacher_profile.detach())

# 3.3 物品级蒸馏损失

pref_distill_loss = self.distill_loss_fn.compute_distill_loss(

student_pref_pred, teacher_preference.detach()

)

# 3.4 动态更新β（解决尺度不平衡）

beta = self.distill_loss_fn.update_beta(epoch, total_epochs=100)

# ---------- Step 4: 总损失 = 模型损失 + β * 蒸馏损失 ----------

total_loss = model_loss + beta * (profile_loss + pref_distill_loss)

# ---------- Step 5: 反向传播 ----------

self.optimizer.zero_grad()

total_loss.backward()

self.optimizer.step()

return {

'total_loss': total_loss.item(),

'model_loss': model_loss.item(),

'distill_loss': (profile_loss + pref_distill_loss).item(),

'beta': beta.item()

}

❓ 常见问题

Q：为什么需要动态缩放因子β？

A：因为蒸馏损失Ldistill通常比模型损失Lmodel小几个数量级，直接相加会导致优化器主要关注主损失而忽视蒸馏知识。动态β通过自适应调整确保两类损失在优化中保持适当比例，类似课程学习让模型在不同阶段侧重不同目标。

Q：这个方法相比直接用LLM做推荐有什么优势？

A：最直接的优势是推理效率和部署成本。蒸馏后的学生模型是轻量级Transformer，毫秒级响应；而每次请求LLM可能需要秒级且成本高昂。此外学生模型可离线批量推理，适合高并发推荐场景。

Q：用户画像推断的质量如何保证？

A：LLM基于用户历史行为序列生成画像时，通过设计特定的Prompt引导其关注偏好、区分高低评分物品等关键特征。实验中Beauty数据集的案例表明推断画像具有较好的可解释性和准确性。
