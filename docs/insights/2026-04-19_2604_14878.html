## 📖 背景与动机

生成式检索（Generative Retrieval, GR）通过下一token预测（NTP）范式为推荐系统带来新思路。然而，将GR应用于工业级大规模推荐系统面临三重挑战：首先是分页请求导致的一致性问题——同一请求的多次调用可能因模型输入相同而产生不一致的输出；其次是计算成本问题——将长用户行为序列编码为多token的语义ID会带来巨大的计算开销；第三是偏好对齐问题——如何让生成式策略与细粒度的用户偏好信号有效对齐。现有方法主要在学术数据集上验证，缺乏工业级部署经验。GenRec正是针对这些挑战提出的完整解决方案。


**关键要点：**

- 生成式检索通过NTP范式统一了推荐系统的编码和解码过程
- 工业系统需处理大规模用户行为序列和实时响应需求
- 分页机制、计算成本、偏好对齐是落地的三大核心障碍

## 💡 核心方法

### 方法概述

GenRec采用单一decoder-only架构，通过语义ID量化高维商品，利用页面级NTP任务提供更密集的梯度信号，并设计非对称Token Merger实现高效编码。


### 详细设计

GenRec的架构包含三个核心设计。首先是语义ID生成：利用量化方法将高维商品embedding映射为离散的语义ID序列，实现商品的紧凑表示。其次是Page-wise NTP训练任务：传统NTP对每个交互商品单独监督，导致一对多歧义问题；Page-wise NTP以整个交互页面为监督单元，聚合多个商品的监督信号，提供更密集的梯度信息，加速收敛并提升效果。第三是Token Merger机制：采用非对称设计，编码端使用轻量级线性层将多token序列压缩为固定长度表示，解码端保持完整的语义ID生成能力，在效率和效果间取得平衡。训练过程中，模型学习预测用户在给定页面上下文后的下一个交互商品对应的语义ID序列。

### 🔧 关键组件

| 组件 | 说明 |
|------|------|
| Semantic ID Encoder | 将高维商品特征（如图片、文本、用户行为统计）通过量化器映射为离散的语义ID序列，支持语义相近商品拥有相似ID前缀 |
| Token Merger | 非对称线性压缩模块，编码时将长序列压缩为固定长度表示，解码时保持逐token生成能力，平衡计算效率和表达能力 |
| Page-wise NTP Loss | 以交互页面为单位的监督信号，计算整个页面的预测损失而非逐商品损失，提供更密集的梯度更新 |

### 💻 代码示例

import torch
import torch.nn as nn

# ==================== 1. 语义ID生成（量化映射）====================
class SemanticIDGenerator(nn.Module):
    """利用量化方法将商品embedding映射为离散语义ID"""
    
    def __init__(self, embedding_dim, num_centroids):
        super().__init__()
        self.codebook = nn.Embedding(num_centroids, embedding_dim)  # 码本
        self.projection = nn.Linear(embedding_dim, num_centroids)   # 映射层
    
    def quantize(self, embeddings):
        """将embedding量化到最近的语义ID"""
        # 计算与所有码本向量的距离
        logits = self.projection(embeddings)
        semantic_ids = torch.argmax(logits, dim=-1)  # 离散语义ID
        # 获取对应的码字（量化后的表示）
        quantized = self.codebook(semantic_ids)
        return semantic_ids, quantized
    
    def forward(self, embeddings):
        return self.quantize(embeddings)


# ==================== 2. Token Merger（编码端压缩）====================
class TokenMerger(nn.Module):
    """非对称设计：编码端压缩，解码端保持完整"""
    
    def __init__(self, num_tokens, hidden_dim, compressed_len=4):
        super().__init__()
        self.compressed_len = compressed_len
        # 编码端：轻量级线性压缩
        self.encoder_fusion = nn.Linear(num_tokens, compressed_len)
        # 解码端：保持原始token数
        self.decoder_expand = nn.Linear(compressed_len, num_tokens)
    
    def encode(self, multi_token_seq):
        """编码端：多token -> 固定长度"""
        # multi_token_seq: [batch, num_tokens, hidden_dim]
        # 先融合token维度，再压缩
        fused = torch.mean(multi_token_seq, dim=1)  # 平均融合
        compressed = self.encoder_fusion(fused)      # 压缩到固定长度
        return compressed
    
    def decode(self, compressed_seq):
        """解码端：恢复到完整语义ID生成能力"""
        expanded = self.decoder_expand(compressed_seq)  # 恢复长度
        return expanded
    
    def forward(self, multi_token_seq):
        compressed = self.encode(multi_token_seq)
        recovered = self.decode(compressed)
        return compressed, recovered


# ==================== 3. Page-wise NTP训练 ========================
class PageWiseNTPLoss(nn.Module):
    """Page-wise Next Token Prediction：以页面为监督单元"""
    
    def __init__(self, model):
        super().__init__()
        self.model = model
    
    def page_level_loss(self, page_embeddings, next_item_embedding):
        """
        页面级损失：聚合多个商品的监督信号
        解决一对多歧义问题，提供更密集梯度
        """
        # 获取页面所有商品的语义ID
        page_semantic_ids, page_quantized = self.model.quantize(page_embeddings)
        next_semantic_id, next_quantized = self.model.quantize(next_item_embedding)
        
        # 页面级对比损失：预测下一个商品的语义ID序列
        # 方式1：聚合页面内所有商品信息
        page_aggregated = torch.mean(page_quantized, dim=0)  # [hidden_dim]
        
        # 方式2：使用注意力加权聚合
        # page_aggregated = self.attention_pooling(page_quantized)
        
        # 预测下一个商品
        logits = self.model.projection(page_aggregated.unsqueeze(0))
        target_id = next_semantic_id  # 单一监督信号
        
        # 交叉熵损失
        loss = nn.functional.cross_entropy(logits, target_id)
        return loss
    
    def forward(self, page_batch, next_item_batch):
        """page_batch: 一个交互页面内的所有商品embeddings"""
        total_loss = 0
        for page_embeddings in page_batch:
            # 随机选择下一个交互商品作为预测目标
            next_item = next_item_batch  # [1, embedding_dim]
            loss = self.page_level_loss(page_embeddings, next_item)
            total_loss += loss
        return total_loss / len(page_batch)


# ==================== 4. GenRec 完整模型 ========================
class GenRecModel(nn.Module):
    """GenRec：整合三个核心设计"""
    
    def __init__(self, embedding_dim=64, num_items=10000, 
                 num_centroids=1024, num_tokens=8, compressed_len=4):
        super().__init__()
        # 商品embedding
        self.item_embedding = nn.Embedding(num_items, embedding_dim)
        # 语义ID生成器
        self.semantic_id_gen = SemanticIDGenerator(embedding_dim, num_centroids)
        # Token Merger
        self.token_merger = TokenMerger(num_tokens, embedding_dim, compressed_len)
        # 预测头
        self.predict_head = nn.Linear(embedding_dim, num_centroids)
    
    def forward(self, page_item_ids, target_item_id):
        """
        Args:
            page_item_ids: 页面内交互商品ID [batch, seq_len]
            target_item_id: 目标商品ID [batch]
        """
        # Step 1: 获取embedding并生成语义ID
        page_embeddings = self.item_embedding(page_item_ids)
        semantic_ids, quantized = self.semantic_id_gen(page_embeddings)
        
        # Step 2: Token Merger压缩
        compressed, recovered = self.token_merger(page_embeddings)
        
        # Step 3: 页面级预测（使用压缩表示）
        page_repr = compressed.mean(dim=0)  # [hidden_dim]
        logits = self.predict_head(page_repr.unsqueeze(0))
        
        # 获取目标语义ID
        target_emb = self.item_embedding(target_item_id)
        target_sem_id, _ = self.semantic_id_gen(target_emb)
        
        loss = nn.functional.cross_entropy(logits, target_sem_id)
        return loss, semantic_ids

## ❓ 常见问题

**Q：Page-wise NTP相比标准NTP的优势是什么？**

A：标准NTP对每个交互商品单独监督，当相同输入可能对应多个合理输出时存在一对多歧义。Page-wise NTP以整个页面为监督单元，聚合多个商品的监督信号，提供更密集的梯度更新，加速收敛并提升模型对用户偏好模式的捕获能力。


**Q：Token Merger如何实现效率和效果的平衡？**

A：采用非对称设计：编码端使用轻量级线性层将用户长行为序列压缩为固定长度表示，降低计算复杂度；解码端保持完整的逐token语义ID生成能力，确保推荐结果的可解释性和精确性。


**Q：语义ID相比传统商品ID有什么优势？**

A：语义ID通过量化高维embedding得到，相近商品拥有相似ID前缀或结构，使模型能够学习商品的语义聚类信息，提升泛化能力，尤其对新商品和冷启动场景更友好。
