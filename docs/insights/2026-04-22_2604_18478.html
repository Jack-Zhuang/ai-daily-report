## 📖 背景与动机

当前大语言模型驱动的聊天机器人在处理跨会话记忆时面临根本性瓶颈。基于扁平向量存储的RAG系统将知识碎片化为无结构chunk，导致三个核心问题：一是事实被割裂在独立片段中，丢失上下文关联；二是跨会话时实体身份无法保持一致；三是缺乏对知识更新（supersession）和矛盾检测的一等支持。虽然Graphiti、Memento、Hydra DB等双时态知识图谱系统引入了类型化边和有效时间元数据来部分缓解问题，但这些图结构本身仍是扁平的——缺乏递归组合机制、节点无内容寻址约束、边类型只是简单标签而非有行为定义的实体。WorldDB正是在这一背景下提出，试图从根本上重新设计记忆引擎的架构范式。


**关键要点：**

- 现有RAG系统基于扁平向量存储导致知识碎片化和上下文丢失
- 双时态知识图谱虽引入时间维度，但图结构本身仍缺乏递归组合能力
- AI Agent从闲聊机器人演进为长时运行系统急需结构化持久记忆能力
- Agent需要理解记忆间的语义层级关系、矛盾检测和更新协调机制

## 💡 核心方法

### 方法概述

WorldDB提出以World为基本单元的递归图结构，每个World既是容器（包含子图、本体scope、组合嵌入）又是内容寻址的不可变实体。系统通过Handler统一入口进行所有写入操作，实现本体感知的写入时协调。


### 详细设计

WorldDB的核心设计围绕三个非标准承诺展开。首先是World抽象：每个节点不是简单实体而是一个递归容器，容器内可包含子图（child worlds）、边集合以及与自身本体scope对齐的组合嵌入。这种设计允许系统表达「会议中的讨论→讨论中的观点→观点的论据」这样的嵌套层级结构。其次是内容寻址与不可变性：任何节点的编辑都会生成新哈希并向上传播到所有祖先节点，形成类Merkle树结构。这确保了历史可追溯、变更可审计，同时天然支持版本分支和并行编辑场景。第三是Handler Dispatch统一入口：所有写入必须经过Handler而非直接追加，这种设计将协调逻辑（关闭有效性区间、创建派生节点、触发合并）封装在Handler层，确保写入时的语义一致性。Ontology-Aware意味着World携带自身本体定义，Handler在协调时能够理解实体类型约束和关系语义规则，而非仅做字符串级别的匹配。


### 🔧 关键组件

| 组件 | 说明 |
|------|------|
| World容器 | 递归嵌套的知识单元，包含内部子图、本体scope边界和组合嵌入。支持leaf world（空子图）和composite world（非空子图）两种形态，可表达任意深度的语义层级结构。 |
| Handler Dispatch机制 | 系统唯一写入入口，将所有操作（创建、更新、协调、合并）路由到对应的Handler处理器。确保任何状态变更都经过语义协调层，而非裸的追加操作。 |
| Content-Addressed Storage | 基于Merkle树的内容寻址存储引擎。节点不可变、哈希级联、版本分支支持，实现变更可审计和历史可回溯。 |
| Ontology Reconciliation | 本体感知协调器，在写入时检测类型约束冲突、关系一致性、矛盾事实等问题，支持派生节点生成和有效性区间管理。 |
| Composed Embedding | 组合嵌入生成器，从World内部子图结构递归聚合语义信息，生成与本体scope对齐的向量表示，支持结构感知的相似度检索。 |
