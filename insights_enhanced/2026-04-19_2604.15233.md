## 📖 背景与动机

随着大语言模型的兴起，NL2SQL系统成为热点研究方向，旨在让用户通过自然语言查询数据库。然而，现实场景远比实验室假设复杂：用户查询往往跨越多个数据库，需要多轮迭代澄清，还可能涉及外部常识知识。传统NL2SQL的封闭世界假设——即所有答案都存在于单一数据库中——在企业实际应用中严重失效。Blue项目团队观察到，企业级数据应用需要处理来自CRM、ERP、日志系统、外部API等多种来源的结构化和非结构化数据，现有技术难以统一管理这些异构资源。因此，团队提出数据智能层（Data Intelligence Layer）的概念，作为连接用户意图与多源数据的语义桥梁。


**关键要点：**

- NL2SQL系统在真实场景中面临三大挑战：多轮对话、多数据源、常识依赖
- 企业数据环境具有天然的异构性和分布性特征
- 需要一个统一抽象层来弥合用户意图与可用信息之间的语义鸿沟

## 💡 核心方法

### 方法概述

DIL采用注册表（Registry）机制作为核心抽象，将数据源、操作符、元数据统一注册管理，通过Agent编排实现跨源查询处理和流式数据支持。


### 详细设计

DIL的架构设计围绕"注册表"这一核心概念展开。数据注册表（Data Registry）负责管理所有可用数据源的元信息，包括数据schema、访问接口、数据血缘等关键属性，支持结构化和非结构化数据的统一描述。操作符注册表（Operator Registry）则定义了可作用于数据的各类操作，如SQL查询、向量检索、RAG增强、API调用等。Agent作为编排层，通过注册表发现所需的数据和操作符，动态组装查询计划。在执行层面，DIL支持流式数据处理，能够处理实时数据管道而非仅限静态数据库查询。对于跨源查询场景，DIL首先进行语义理解，将用户意图分解为多个子查询，然后根据数据注册表信息将子查询路由到相应数据源，最后合并结果并通过自然语言返回给用户。整个流程体现了"数据即服务"的理念，将复杂的数据集成细节对上层应用透明化。

### 🔧 关键组件

| 组件 | 说明 |
|------|------|
| 数据注册表（Data Registry） | 统一管理多源数据的元信息，提供数据发现、schema管理、血缘追踪能力，支持结构化和非结构化数据的一致性抽象 |
| 操作符注册表（Operator Registry） | 定义数据处理操作的注册与发现机制，包括SQL操作符、向量检索、RAG组件、API调用等多种能力的统一管理 |
| Agent编排引擎（Agent Orchestration） | 负责任务分解、动态路由和执行计划生成，根据用户意图调用相应数据源和操作符组合 |
| 流式处理层（Streaming Layer） | 支持实时数据的订阅、处理和增量查询，而非仅处理静态数据库快照 |

from abc import ABC, abstractmethod
from typing import List, Dict, Any

# ---------------------
# 1. 数据注册表 (Data Registry)
# ---------------------
class DataSource:
    """数据源元信息"""
    def __init__(self, name: str, schema: Dict, source_type: str):
        self.name = name
        self.schema = schema          # 数据schema
        self.source_type = source_type  # sql/nosql/vector/api
        self.access_interface = None  # 访问接口

class DataRegistry:
    """数据注册表 - 管理所有数据源元信息"""
    def __init__(self):
        self._sources: Dict[str, DataSource] = {}
    
    def register(self, source: DataSource):
        """注册数据源"""
        self._sources[source.name] = source
    
    def discover(self, query_hint: str) -> List[DataSource]:
        """根据语义发现相关数据源（伪代码）"""
        # 实际应基于语义匹配/血缘分析
        return list(self._sources.values())


# ---------------------
# 2. 操作符注册表 (Operator Registry)
# ---------------------
class Operator(ABC):
    """操作符基类"""
    @abstractmethod
    def execute(self, data, params: Dict) -> Any:
        pass

class SQLOperator(Operator):
    """SQL查询操作符"""
    def execute(self, data, params: Dict) -> Any:
        # 伪代码：实际执行SQL
        print(f"[SQL] 执行查询: {params.get('query')}")
        return "SQL结果"

class VectorSearchOperator(Operator):
    """向量检索操作符"""
    def execute(self, data, params: Dict) -> Any:
        print(f"[Vector] 检索向量: {params.get('embedding')}")
        return "向量检索结果"

class RAGOperator(Operator):
    """RAG增强操作符"""
    def execute(self, data, params: Dict) -> Any:
        print(f"[RAG] RAG增强上下文")
        return "RAG增强结果"

class OperatorRegistry:
    """操作符注册表 - 管理所有可用操作"""
    def __init__(self):
        self._operators = {
            "sql": SQLOperator(),
            "vector_search": VectorSearchOperator(),
            "rag": RAGOperator(),
        }
    
    def get(self, op_type: str) -> Operator:
        """获取操作符"""
        return self._operators.get(op_type)


# ---------------------
# 3. Agent 编排层
# ---------------------
class DILAgent:
    """DIL Agent - 编排层核心"""
    def __init__(self, data_registry: DataRegistry, op_registry: OperatorRegistry):
        self.data_registry = data_registry
        self.op_registry = op_registry
    
    def understand_intent(self, user_query: str) -> List[Dict]:
        """
        语义理解 - 分解用户意图为子查询
        实际应使用 LLM 进行意图识别
        """
        # 伪代码：解析为多个子查询
        sub_queries = [
            {"type": "sql", "target": "用户表", "intent": "查询用户信息"},
            {"type": "vector_search", "target": "知识库", "intent": "检索相关知识"},
        ]
        print(f"[Agent] 用户意图分解: {sub_queries}")
        return sub_queries
    
    def route_to_source(self, sub_query: Dict) -> Any:
        """根据注册表信息路由到对应数据源"""
        # 1. 从数据注册表发现数据源
        sources = self.data_registry.discover(sub_query["target"])
        
        # 2. 获取操作符
        operator = self.op_registry.get(sub_query["type"])
        
        # 3. 执行操作
        result = operator.execute(sources, sub_query)
        return result
    
    def merge_results(self, results: List[Any]) -> str:
        """合并多源结果并生成自然语言响应"""
        # 伪代码：实际应做结果对齐/去重
        merged = "【合并结果】\n" + "\n".join(results)
        return merged
    
    def query(self, user_query: str) -> str:
        """
        完整查询流程：理解 → 路由 → 执行 → 合并
        """
        print(f"\n{'='*50}")
        print(f"[Query] 用户输入: {user_query}")
        
        # Step 1: 语义理解
        sub_queries = self.understand_intent(user_query)
        
        # Step 2: 路由执行
        results = []
        for sq in sub_queries:
            result = self.route_to_source(sq)
            results.append(result)
        
        # Step 3: 合并返回
        response = self.merge_results(results)
        print(f"{'='*50}\n")
        return response

**Q：DIL与传统数据湖/数据仓库方案有何本质区别？**
